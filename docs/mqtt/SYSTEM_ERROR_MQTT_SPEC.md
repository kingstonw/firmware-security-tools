# FC75 System Error MQTT Specification

This document describes how FC75 publishes error and diagnostic events to MQTT using the system error topic.

It is intended to guide another ESP-IDF project so it can generate compatible error-reporting messages.

---

## 1. Purpose

The system error topic is used to report buffered device errors and diagnostics, including:

- AWS publish/connect failures
- OTA failure paths
- SPIFFS configuration and crypto failures
- Wi-Fi connection/reconnect failures
- BLE provisioning errors
- STM32 transfer failure states
- boot-time restart marker

This is a buffered telemetry topic, not a command topic.

---

## 2. MQTT Topics

### Production

```text
fc75/tx/system/error
```

### Development

```text
fc75/dev/system/error
```

Source constants:

- `config::PATHS::AWS_ERROR = "fc75/tx/system/error"`
- `config::PATHS::DEV_ERROR = "fc75/dev/system/error"`

---

## 3. High-Level Flow

```text
Any module calls Logger::log(LOGGER_ERROR, hexCode, data)
  -> logger writes one record into /error.log
  -> TELEMETRY_scheduler()
  -> case ERROR
  -> RECORD_sendAWS("/error.log", "ERRORS", ...)
  -> AWS_sendEvents(...)
  -> MQTT publish to fc75/tx/system/error or fc75/dev/system/error
  -> on publish success, delete the sent records from /error.log
```

Important behavior:

- Error records are generated by many modules across the project.
- Records are stored locally first in `/error.log`.
- Upload is attempted periodically.
- On publish success, the sent head records are deleted.
- On publish failure, the records remain for retry later.

---

## 4. Publish Preconditions

System error publish is attempted only when all of the following are true:

1. `TELEMETRY_scheduler(sendOnline)` is running.
2. No UI button is currently pressed.
3. `gSYSTEM_WIFI != 0`.
4. `wifi.isConnected()` is true.
5. The scheduler reaches the `ERROR` telemetry slot.

---

## 5. Publish Period

The error upload interval is:

```text
ERROR_SEND_INTERVAL_MS = 70000 ms = 1 minute 10 seconds
```

The scheduler processes one telemetry type per call, so actual send time is approximately periodic.

---

## 6. MQTT Payload Format

The outer MQTT JSON payload is created by `AWS_sendEvents()`.

### JSON schema

```json
{
  "mac": "30:ED:A0:15:6D:A4",
  "type": "ERRORS",
  "timestamp": "2026-04-07T12:34:56Z",
  "mode": "DEV",
  "events": "2026-04-07T12:34:40Z|0x13||1.0.60|851968|1540096"
}
```

### Outer JSON field meanings

| JSON field | Meaning |
|---|---|
| `mac` | Device MAC address |
| `type` | Always `ERRORS` for this topic |
| `timestamp` | Timestamp of this publish attempt |
| `mode` | `DEV` or `PROD` |
| `events` | One or more local error records joined by `~` |

---

## 7. `events` Field Format

The `events` field is a string, not a JSON array.

If one record is sent:

```text
events = "<record1>"
```

If multiple records are sent:

```text
events = "<record1>~<record2>~<record3>"
```

### Batch size

At most 5 error records are sent in one MQTT publish:

```text
ERRORSEND_MAX_RECORDS = 5
```

---

## 8. Single Error Record Format

Each error record inside `events` is stored as:

```text
<record_timestamp>|<hex_code>|<data>|<version>|<spiffsUsed>|<spiffsTotal>
```

This is produced in two steps:

1. `Logger::log(LOGGER_ERROR, hexCode, data)` builds:

```text
<hex_code>|<data>|<version>|<spiffsUsed>|<spiffsTotal>
```

2. `SPIFF_addRecord()` prepends the record timestamp:

```text
<record_timestamp>|<hex_code>|<data>|<version>|<spiffsUsed>|<spiffsTotal>
```

### Example

```text
2026-04-07T12:34:40Z|0x13||1.0.60|851968|1540096
```

### Field meanings

| Position | Field | Meaning |
|---|---|---|
| 1 | `record_timestamp` | UTC ISO 8601 time when the error record was written locally |
| 2 | `hex_code` | Hex identifier from `config::LOGGER::LOG_TABLE` |
| 3 | `data` | Optional detail string, often empty but sometimes dynamic |
| 4 | `version` | ESP32 firmware version |
| 5 | `spiffsUsed` | Used SPIFFS bytes at log time |
| 6 | `spiffsTotal` | Total SPIFFS bytes at log time |

---

## 9. Local Buffering Behavior

Error records are buffered in:

```text
/error.log
```

### Local file limit

```text
ERRORLOG_MAX_RECORDS = 1000
```

If the file is already full, a new record is not written.

### Send/delete behavior

1. Count records in `/error.log`
2. Read up to 5 head records
3. Join them with `~`
4. Publish one MQTT message
5. If publish succeeds, delete exactly those sent records
6. If publish fails, delete nothing

---

## 10. Timestamp Semantics

Two timestamps exist:

### Outer payload timestamp

The JSON `timestamp` field is the send timestamp of the MQTT packet.

### Inner record timestamp

The first field in each error record is when that record was written locally.

These may differ when records are buffered and retried later.

The following table summarizes current `LOGGER_ERROR` producers found in the project.

| Hex | Message in log table | Typical source |
|---|---|---|

| `0x0F` | `[OTA] Update.end failed.` | ESP32 OTA finalize failure |
| `0x13` | `[AWS] AWS connection failed` | AWS connection exhausted retries |
| `0x14` | `[AWS] publish failed` | MQTT publish/truncation/response publish failure |
| `0x15` | `[SPIFF] Invalid AWS config.` | invalid AWS config content |
| `0x16` | `[SPIFF] Invalid or incomplete AWS certificates.` | invalid certs or parse failures in some SPIFF paths |
| `0x17` | `[SPIFF] WiFi config read failed or file empty` | Wi-Fi config load failed |
| `0x18` | `[SPIFF] WiFi config JSON parse failed` | Wi-Fi config JSON parse failure |
| `0x19` | `[SPIFF] WiFi config missing required fields` | Wi-Fi config missing fields |
| `0x1B` | `[SPIFF] WiFi config missing required fields` | alternate Wi-Fi credentials loader missing fields |
| `0x1C` | `[SPIFF] CA cert missing or invalid` | CA or device cert missing/invalid paths |
| `0x1D` | `[SPIFF] Key missing or invalid` | private key missing/invalid |
| `0x1E` | `[SPIFF] AWS config read failed or file empty` | AWS config read failure |
| `0x1F` | `[SPIFF] failed to deserialize the json file` | AWS config JSON parse failure |
| `0x20` | `[SPIFF] One or more AWS credentials are missing or invalid` | one or more AWS credential files invalid |
| `0x21` | `[SPIFF] Error decrypting file` | decrypting encrypted SPIFF file failed |
| `0x22` | `[SPIFF] Encryption failed` | could not open file for encrypted save |
| `0x23` | `[SPIFF] Failed to write all bytes` | encrypted file write incomplete |
| `0x25` | `[WIFI] SSID or Password failed` | Wi-Fi connect called with empty SSID/password |
| `0x29` | `[WIFI] Reconnect failed` | Wi-Fi reconnect attempt failed |
| `0x2D` | blank in log table | encryption returned empty output during saveEncrypted |
| `0x2F` | blank in log table | alternate Wi-Fi config read/decrypt path failed |
| `0x30` | blank in log table | STM32 payload transfer state machine entered failure branch |

### Important note about blank descriptions

Some hex codes are actively used but their human-readable string in `config::LOGGER::LOG_TABLE` is blank.
For these codes, cloud consumers should treat the hex code itself as authoritative.

Currently observed blank-but-used codes:

- `0x2D`
- `0x2F`
- `0x30`

## 12. The `data` Field Behavior

The `data` field is often an empty string, so records may look like:

```text
2026-04-07T12:34:40Z|0x0D||1.0.60|851968|1540096
```

However, sometimes `data` carries useful dynamic content.

Examples:

1. MQTT publish failure may store the failed topic path in `data`
2. Payload truncation paths may store a descriptive text string

So parsers should never assume `data` is non-empty, and should preserve empty-field position.

## 14. ESP-IDF Compatibility Guidance

To generate compatible `fc75/tx/system/error` messages in an ESP-IDF project:

1. Maintain a persistent local error queue equivalent to `/error.log`
2. Store each error record as:

```text
<timestamp>|<hex>|<data>|<version>|<spiffsUsed>|<spiffsTotal>
```

3. Every 70 seconds, attempt to send up to 5 pending records
4. Join multiple records with `~`
5. Publish JSON:

```json
{
  "mac": "<mac>",
  "type": "ERRORS",
  "timestamp": "<utc_iso8601>",
  "mode": "DEV or PROD",
  "events": "<record1>~<record2>..."
}
```

6. Publish to:

```text
PROD -> fc75/tx/system/error
DEV  -> fc75/dev/system/error
```

7. Delete sent records only on MQTT publish success
8. Preserve records on publish failure

---

## 15. Reference Behavior Summary

The FC75-compatible error reporting model is:

- write every error locally first
- publish buffered error batches approximately every 1 minute 10 seconds
- send at most 5 records per MQTT message
- use `events` as a string with `~` between records
- preserve exact `|` field order even when `data` is empty
- delete only after successful publish

That is the wire-level behavior another ESP-IDF project should reproduce.
